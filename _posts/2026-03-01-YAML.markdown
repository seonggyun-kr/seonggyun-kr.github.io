지난 포스팅에서 YAML의 기초를 다뤘다면, 이번에는 **딥러닝(Deep Learning) 실험 환경**을 구축할 때 YAML을 어떻게 활용하여 효율적으로 관리할 수 있는지 알아보겠습니다.

## 1. 하이퍼파라미터 관리의 어려움

딥러닝 모델을 학습시킬 때는 연구자가 직접 결정해야 하는 수많은 **하이퍼파라미터(Hyperparameter)**가 존재합니다. 

* **기본 설정:** 손실 함수(Loss Function)의 종류, 학습률(Learning Rate), 배치 사이즈(Batch Size) 등
* **세부 설정:** 특정 레이어의 가중치 고정(Freezing), 스케줄러(Scheduler) 수치 등

하나의 완성도 있는 모델을 얻기 위해서는 수십 번에서 많게는 수천 번의 실험을 반복해야 합니다. 이때 매번 터미널에서 `argparse`와 같은 인자 분석기를 통해 값을 입력하는 방식은 다음과 같은 한계가 있습니다.

1. **비효율성:** 실험 인자가 많아질수록 명령어가 너무 길어져 관리가 불가능해집니다.
2. **휴먼 에러:** 사람이 직접 입력하다 보니 값을 오타 내거나, 엉뚱한 인자에 값을 넣는 실수가 잦아집니다.
3. **재현성 결여:** 실험이 끝난 후 내가 어떤 설정값으로 이 결과를 냈는지 기록이 남지 않아 다시 재현하기 어렵습니다.

---

## 2. Configuration 파일의 도입

이러한 문제를 방지하기 위해, 한 실험에 사용할 **인자와 해당 값을 하나의 파일에 종합하여 기록**해두는 방식을 사용합니다. 우리는 이러한 파일을 **Config** 또는 **Configuration(설정)** 파일이라고 부릅니다.

보통 프로젝트 레포지토리에 `config/`라는 디렉토리를 만들고, 그 안에 여러 개의 설정 파일을 관리하는 것이 관례입니다. 이때 가장 많이 쓰이는 형식이 바로 **YAML**입니다.

### YAML을 사용하는 이유
* **가독성:** 구조가 계층적이고 직관적이어서 사람이 읽고 수정하기 매우 편합니다.
* **데이터 변환:** Python의 `Dictionary` 타입으로 쉽게 변환되어 코드 내에서 값을 꺼내 쓰기 용이합니다.
* **실험 기록:** 실험 종료 시 사용했던 YAML 파일을 결과 파일(Checkpoints)과 함께 저장하면, 추후 어떤 설정으로 실험했는지 명확하게 확인할 수 있습니다.

---

## 3. 예시: YAML 설정 파일 구조

실제로 딥러닝 프로젝트에서 사용하는 YAML 파일은 대략 다음과 같은 구조를 가집니다.

{% highlight yaml %}
model:
  name: "ResNet50"
  pretrained: true
  num_classes: 10

train:
  epochs: 100
  batch_size: 32
  learning_rate: 0.001
  optimizer: "Adam"

dataset:
  path: "./data/cifar10"
  augmentation: true
{% endhighlight %}

우리가 위와 같은 형식의 `resnet50.yaml`이라는 설정 파일을 만들어 두었다고 가정해 봅시다. 동일한 실험을 수행할 때 터미널에 입력하는 명령어의 차이는 극명합니다.

### 기존 방식: 모든 인자를 터미널에 입력 (CLI Argument)
인자가 늘어날수록 명령어가 한 화면을 넘어갈 정도로 길어집니다. 오타가 날 확률이 매우 높고, 나중에 어떤 설정으로 돌렸는지 기록하기 어렵습니다.

{% highlight bash %}
python train.py --model_name ResNet50 --pretrained True --num_classes 10 --epochs 100 --batch_size 32 --lr 0.001 --optimizer Adam --data_path ./data/cifar10 --aug True
{% endhighlight %}

### Config 파일 활용 방식: 설정 파일 경로를 터미널에 입력 

{% highlight bash %}
python train.py --config resnet50.yaml
{% endhighlight %}

이처럼 설정값을 파일에 미리 정의해두면 명령어는 매우 깔끔해집니다. 어떤 실험인지 파일명만으로도 식별이 가능하며, 설정 파일 자체가 실험의 로그가 됩니다.

## 4. YAML 파일의 확장: 상속(Inheritance)과 Include

실험을 하다 보면 "기존 설정에서 파라미터 한두 개만 바꾸고 싶은데, 그때마다 매번 수십 줄짜리 파일을 새로 만들어야 할까?"라는 고민이 생깁니다. 이때 유용한 것이 바로 **상속(Inheritance)** 구조입니다.

공통적인 설정은 '뼈대(Base)' 파일에 정의하고, 개별 실험에서는 차이가 있는 부분만 작성하여 효율을 극대화할 수 있습니다.

### 뼈대(Base) 파일 예시

실험의 구성 요소를 모듈화하여 관리하면 유지보수가 매우 쉬워집니다. 보통 아래와 같이 기능별로 파일을 분리합니다.

**1. `configs/dataset/cifar10.yml` (데이터셋 공통 설정)**
{% highlight yaml %}
dataset:
  name: "CIFAR10"
  path: "./data/cifar10"
  batch_size: 128
  num_workers: 4
  augmentation:
    resize: 32
    normalize: [[0.4914, 0.4822, 0.4465], [0.2023, 0.1994, 0.2010]]
{% endhighlight %}

**2. `configs/runtime.yml` (실험 환경 공통 설정)**
{% highlight yaml %}
runtime:
  gpu_ids: [0]
  seed: 42
  checkpoint_interval: 10
  log_dir: "./runs/exp_logs"
{% endhighlight %}

### 상속을 통한 실험 파일 구성

실제 실험 파일에서는 위 파일들을 불러오기(Include)만 하면 됩니다. `__include__`와 같은 키워드를 통해 여러 파일을 병합(Merge)하는 방식입니다.

**`configs/exp_resnet50_v1.yaml`**
{% highlight yaml %}
__include__: [
  './dataset/cifar10.yml',
  './runtime.yml',
  './base/optimizer.yml',
  './models/resnet50.yml'
]

# 불러온 설정 중 특정 값만 이번 실험에 맞춰 덮어쓰기(Override)
dataset:
  batch_size: 64              # 메모리 확보를 위해 배치 사이즈 수정

train:
  epochs: 200                 # 더 긴 학습을 위해 에포크 수정
  learning_rate: 0.0005       # 이번 실험을 위한 커스텀 학습률
{% endhighlight %}



### 상속 방식을 사용하면 얻는 이점
* **재사용성 극대화:** 데이터셋 경로 등이 바뀌어도 수십 개의 파일을 수정할 필요 없이 Base 파일 하나만 고치면 모든 실험에 반영됩니다.
* **가독성 확보:** 실험 파일 본문에는 '이번 실험에서 바뀐 핵심 수치'만 남게 되므로, 실험의 의도가 명확히 드러납니다.
* **직관적인 관리:** 프로젝트 규모가 커져도 `dataset`, `model`, `optimizer` 단위로 설정을 모듈화하여 관리할 수 있어 구조가 깔끔해집니다.

## 5. 마치며: 실험 기록으로서의 YAML

결과적으로 딥러닝에서 YAML 설정 파일을 설계하는 것은 단순한 편의를 넘어 **"실험의 재현성"**을 확보하는 핵심 과정입니다. 실험이 끝날 때마다 사용했던 최종 YAML 파일을 결과 폴더(Checkpoints)에 함께 복사해 두는 습관을 들여보세요. 수개월 뒤 다시 코드를 열었을 때, 여러분이 과거에 어떤 고민을 하며 하이퍼파라미터를 조정했는지 알려주는 가장 확실한 이정표가 되어줄 것입니다.
